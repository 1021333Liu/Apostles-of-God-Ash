# 日志碎片系统计划

## 系统定位

日志碎片系统是《神烬使徒》低语田野垂直切片的第二核心循环。

战斗解决“我如何活下来”，日志碎片解决“我为什么要继续进入田野”。玩家击杀小怪、击败精英、通过房间和死亡回收时，都应该得到碎片化叙事材料，并在圣匣中逐步拼出第一段故事。

## MVP 目标

第一版不做复杂推理、不做多结局、不做大图鉴。只做一条清晰闭环：

```text
击杀敌人 -> 掉落日志碎片 -> 拾取 -> 自动归档 -> 圣匣查看 -> 解锁第一段故事
```

## 第一阶段范围

### 1. 掉落

- 空腹者：低概率掉落普通日志碎片。
- 饥民农夫：中概率掉落田地管理相关碎片。
- 饥饿稻草人：中概率掉落旧命令/偷粮者相关碎片。
- 谷仓王：必掉 Boss 记忆碎片。

### 2. 拾取

- 日志碎片作为 `Area2D` 或当前原型中的拾取数据存在。
- 玩家靠近后自动吸收，或按 `E` 收集。
- 拾取时显示短反馈：

```text
样本归档：田边剩粮
```

### 3. 归档

圣匣中显示已收集碎片：

- 敌人身份
- 来源房间
- 文本片段
- 胃囊反应
- 是否属于主线拼图

### 4. 拼接

第一版只做一条主线：

```text
低语田野第一故事：谷仓王为什么献出自己的胃
```

收集到指定数量或指定组合后，圣匣显示完整段落。

## 数据结构建议

先用 JSON 或 `.tres` 都可以。为了方便文案协作，第一版建议用 JSON：

```text
data/log_fragments/low_whispering_field.json
```

建议字段：

```json
{
  "id": "field_empty_01",
  "title": "田边剩粮",
  "source_enemy": "empty",
  "source_room": "field_gate",
  "rarity": "common",
  "main_story_key": "barn_king_stomach",
  "text": "碗是空的，但碗底被舔得很亮。",
  "stomach_reaction": "胃囊轻微收缩，拒绝判断这是不是食物。"
}
```

## UI 设计方向

圣匣日志 UI 不应该像普通菜单。

它应该像一个“活器官档案柜”：

- 左侧：碎片列表，像样本编号。
- 中央：当前碎片文本。
- 右侧：胃囊反应和拼图进度。
- 底部：第一故事的解锁进度。

文字不要太长。每个碎片控制在 1-3 句。

## 文案任务

文案同学优先交付：

- 空腹者碎片 6 条。
- 饥民农夫碎片 6 条。
- 饥饿稻草人碎片 6 条。
- 谷仓王碎片 4 条。
- 第一故事完整段落 1 条。
- 每类碎片的胃囊反应各 3 条。

交付格式见 `08_NARRATIVE_HANDOFF_PROMPT.md`，但要新增 `id`、`source_enemy`、`source_room` 字段。

## 程序任务

建议单独创建：

```text
feature/log-fragment-system
```

首批文件：

- `data/log_fragments/low_whispering_field.json`
- `scripts/systems/log_fragment_database.gd`
- `scripts/systems/log_archive_runtime.gd`
- `scripts/ui/log_archive_panel.gd`

当前原型阶段可以先不拆场景，先把系统跑通；但不要把所有日志数据硬编码进 `scripts/main.gd`。

## 验收标准

- 击杀至少 3 类敌人能获得不同日志碎片。
- 圣匣中能查看已获得碎片。
- 死亡回圣匣后能看到本轮获得内容。
- 收集指定碎片后能解锁“谷仓王为什么献出胃”的第一故事。
- Godot 主场景启动无错误。
- `CHANGELOG.md` 记录系统接入。

## 风险

- 文本过长会拖慢战斗节奏。
- 掉落太随机会导致玩家看不到故事。
- UI 如果太像普通图鉴，会削弱神之胃囊的世界观。
- 如果继续硬编码在 `main.gd`，后续文案扩展会非常痛苦。

## 推荐优先级

P0：

- 数据文件。
- 拾取和归档。
- 圣匣查看 UI。

P1：

- 第一故事拼接。
- 死亡回收总结。
- 稀有碎片提示。

P2：

- 搜索/筛选。
- 多故事线。
- 动态文本组合。
- 日志碎片美术特效。
